Learn Java Essentials from One Class

Author: Chen Wahng Date: Jul 06, 2019  15:30:42 Category: Java

To understand this blog, you have to be familiar with some basic programming concepts, such as variables, constants, operators, control flow, etc. This blog mainly comes from my notes of the brilliant Core Java series Core Java by Cay S. Horstmann and Gray Cornell. If you need thorough and systematic learning of Java, I recommend you to read those books, at least Volume I, Fundamentals. Apparently, I’m not a Java expert, and I make easy mistakes, besides, I’m not a native speaker, so some of my statement may be buggy. Sincerely, any comments and advice are highly welcome.

[1] Package Statement

  • Put the line package [package name] at the beginning of your file (below shebang). This command declares this class belongs to which package.
  • Package is commonly named with the reverse order of a public domain, so they won’t collide with others.
  • Package name also decides the directory structure of your project. See another blog Maven Tutorial #groupID for more details.

[2] Import Statements

  • Put the import statements right below your package statement.
  • Each of them imports one external class to your current class. You can access all of their public features (that is to say, fields and methods).
  • Import static features with the code import static [package name.ClassName.featureName or *];, so you can use the function without the prefix. Another common example import static java.lang.Math.*;.

[3] Class Declaration

  • public: One source file (.java file) can only have one public class. The reason is, Java Virtual Machine (JVM) relies on the binary file (‘.class’ file, generated from .java file the with same name) name to locate a class. If you put more than one public class inside a source file, JVM will mess up them.
  • Stack<Item>: The class name. A class name always starts with an upper case word and follows the CamelRule. The diamond <> represents another heavy topic in Java, Generics, which I will talk about it later in another blog. In this blog, I will simply strip the generics to make things easy.
  • implements: This keyword means Iterable is an interface. Declare an interface with public interface Iterable. Although an interface looks like a class, it’s rather a series of requirements for the class that intends to implement it. All methods in an interface are automatically public and all fields must be public, static, final, as an interface cannot have instance fields. One class can implement many interfaces, and it has to develop all interface methods or declare them to be abstract. An interface cannot be initialized, that is to say, you cannot new an instance of it.
  • Abstract class: If a class leave a part of method implementations empty, this class is an abstract class. More specifically, an abstract class implements part of its methods while leaving the others empty (those are so-called abstract method). This differentiates it from an interface. However, similar to an interface, an abstract class cannot be initialized. As you can see from the above code snippets, an abstract method must be specified with an abstract keyword. If you inherit an abstract class, you need to implement the abstract methods in it. If all of them are implemented, the subclass is no longer abstract, otherwise, it’s still abstract and the keyword abstract in the class declaration is still necessary.
  • extends is another common keyword you can find in the class declaration, it states an inheritance relationship between two classes. If public class A extends B, then A inherits all public features from B. A is called the subclass of B, and on the contrary, B is the superclass of A. One class can only have one superclass, in order to avoid the diamond problem. In A‘s body, super refers to its superclass, so you can call B‘s constructor with super() in A‘s constructor to quickly initialize some fields. Btw, this refers to the class itself in any class’s body. A could reimplement some methods in B, this is called override.

[4] Access Control

  • public: open to everything.
  • private: open to everything inside its class. An instance has access to private features of other instances of the same class.
  • blank: open to everything inside its package.
  • protected: open to everything inside its class and all subclasses, including ones from other packages. Declare a feature as protected breaks the safe encapsulation, which is un-recommended.
  • static and final: These two keywords are not in the scope of access control, but they usually come with those access control keywords, so I talk about them in this section. static declares a class feature, it means this feature is accessible even without an instance. So a static method can only accept static fields as arguments. Call a static feature by ClassName.featureName, instead of the common way instanceName.featureName. A class only have one copy of their static features in memory, and all instances share them. We will talk about static class later. Applying final to a field denotes a constant field. If it refers to a primitive type (they are boolean, byte, short, char, int, long, float, double in Java), it means the value of this field can not be changed. If it’s an Object, the value of this Object is still changeable, but you can not make this field refer to another thing. In short, the assignment statement is prohibited for a final field. A final class can not be inherited, and a final method can not be overridden. You can use the combo of them static final as a prefix.

[5] Constructor

  • Unlike a method, a constructor doesn’t have a return type.
  • The Stack() is called default constructor, even if you leave it blank, JVM still assign all uninitialized fields with their default values, 0 for primitive numbers and char, false for boolean, null for Objects.
  • The Stack(Item) is a parameterized constructor.
  • You can use any other constructors, such as this() or super(), inside a constructor. First of all, make sure such a constructor exists. Also, put any super() constructors before the this() ones.

[6] Getter and Setter

Usually, we set all class fields as private, to avoid accidentally change and malicious access from other classes. Sometimes, it’s necessary to change or get a field from outside, so Java designs setter and getter. The size() function in this class is a getter. Sometimes we name them with the field name, e.g. setN() and getN().

[7] Equality Test

  • In total, there are 2 kinds of equality test, identity-oriented (with calling equals() method) and value-oriented (with equal sign ==). The former one tests whether 2 variables occupy the same memory address, while the later one focuses on comparing their value.
  • Primitive types: all primitive types are value-oriented compared. Be careful about equality between double types, sometimes the precision difference caused by hardware limitation results in an invalid program.
  • String: all String are immutable Objects in Java, it means we can not change the content of a String once it’s created. Treat each string as an object, use == to test the memory address equality, and use equals() for content equality test.

[8] Documentation

Documentation is extremely crucial to code management, inheritance and update. You can use an inline documentation (// follows documentation content) to explain simple points, while use a block documentation (as showed in the above code snippets) to illustrate a class, a method, or any other obscure points.
Roughly speaking, there are 3 kinds of documentation,

  1. Class Documentation: usually comes as a block documentation, right above the class to be explained. State the authorities, objectives and functionality of this class briefly.

  2. Method Documentation: block documentation as well. Below is the structure of a method documentation.

  3. Class Field Documentation or Code Block Documentation: explain the function of a class field, or the logic of a code block with inline documentation, in a pithy style.

Java documentation supports a variant of tags and html style. Besides what you have seen before,

  • @author [author name]
  • {@code [code snippets]}
  • italic, emphasis, , etc.

[9] toString() Method

A class implement the toString() method is printable. In the code System.out.print(objectName), the JVM will automatically call the objectName.toString() method of this object, and replace the objectName with the return value of objectName.toString(). So this method should return a String contains essential values in this Object. Actually, all class should implement this method for convenient debug.

[10] Static Inner Class

Sometimes we need to hide one class inside another to either avoid name collision or benefits package management. Static inner class, or nested class comes into being under such requirement.

  • Only inner class can be declared as static.
  • Declare a static inner class as public, private, protected, or leave it blank, then you can call it through OuterClassName.InnerClassName under the access rule defined by the keyword.
  • Static inner class act exactly like a top-level class, so it’s not a member of the outer class. For example, static inner class can have static features. Also, it has no privilege about accessing to outer class’s feature.
  • An interface’s inner class are automatically static and public.
