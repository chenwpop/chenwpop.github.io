Learn Java Essentials from One Class

Author: Chen Wahng Date: Jul 6, 2019  15:30:42 Category: Java

To understand this blog, you have to be familiar with some basic programming concepts, such as variables, constants, operators, control flow, etc. This blog mainly comes from my notes of the brilliant Core Java series Core Java by Cay S. Horstmann and Gray Cornell. If you need thorough and systematic learning of Java, I recommend you to read those books, at least Volume I, Fundamentals. Apparently, I’m not a Java expert, and I make easy mistakes, besides, I’m not a native speaker, so some of my statement may be buggy. Sincerely, any comments and advice are highly welcome.

[1] Package Statement

  • Put the line package [package name] at the beginning of your file (below shebang). This command declares this class belongs to which package.
  • Package is commonly named with the reverse order of a public domain, so they won’t collide with others.
  • Package name also decides the directory structure of your project. See another blog Maven Tutorial #groupID for more details.

[2] Import Statements

  • Put the import statements right below your package statement.
  • Each of them imports one external class to your current class. You can access all of their public features (that is to say, fields and methods).
  • Import static features with the code import static [package name.ClassName.featureName or *];, so you can use the function without the prefix. Another common example import static java.lang.Math.*;.

[3] Class Header

public

One source file (.java file) can only have one public class. The reason is, Java Virtual Machine (JVM) relies on the binary file (.class file, generated from .java file the with same name) name to locate a class. If you put more than one public class inside a source file, JVM will mess up them.

Stack<Item>

  • The class name. A class name always starts with an upper case word and follows the CamelRule.
  • The diamond <> represents another heavy topic in Java, Generics, which I will talk about it later in another blog. In this blog, I will simply strip the generics to make things easy.

implements

  • This keyword means Iterable is an interface. Declare an interface with public interface Iterable {...}.
  • Although an interface looks like a class, it’s rather a series of requirements for the class that intends to implement it.
  • All methods in an interface are automatically public and all fields must be public, static, final, as an interface cannot have instance fields.
  • One class can implement many interfaces, and it has to develop all interface methods or declare them to be abstract. - An interface cannot be initialized, that is to say, you cannot new an instance of it.

Abstract Class

  • If a class leave a part of method implementations empty, this class is an abstract class. More specifically, an abstract class implements part of its methods while leaving the others empty (those are so-called abstract method). This differentiates it from an interface.
  • Similar to an interface, an abstract class cannot be initialized.
  • As you can see from the code snippets below, an abstract method must be specified with an abstract keyword. If you inherit an abstract class, you need to implement the abstract methods in it. If all of them are implemented, the subclass is no longer abstract, otherwise, it’s still abstract and the keyword abstract in the class header is still necessary.

extends

  • This is another common keyword you can find in the class header, it states an inheritance relationship between two classes. If public class A extends B, then A inherits all public features from B. A is called the subclass of B, and on the contrary, B is the superclass of A.
  • One class can only have one superclass, in order to avoid the diamond problem.
  • If you reimplement a superclass’s method in the subclass, that is to say, you override it, then the method in subclass can not have narrower access control keyword than those in the superclass.
  • In A‘s body, super refers to its superclass, so you can call B‘s constructor with super() in A‘s constructor to quickly initialize some fields.
  • BTW, this refers to the class itself in any class’s body. A could reimplement some methods in B, this is called override.

[4] Access Control

Public, Private, Protected, and the default

  • public: open to everything.
  • private: open to everything inside its class. An instance has access to private features of other instances of the same class.
  • blank (package private, the default case): open to everything inside its package.
  • protected: open to everything inside its class and all subclasses, including ones from other packages. Declare a feature as protected breaks the safe encapsulation, which is un-recommended.

static and final

  • These two keywords are not in the scope of access control, but they usually come with those access control keywords, so I talk about them in this section.
  • static declares a class feature, it means this feature is accessible even without an instance. So a static method can only accept static fields as arguments.
  • Call a static feature by ClassName.featureName, instead of the common way instanceName.featureName.
  • A class only have one copy of their static features in memory, and all instances share them.
  • Applying final to a field denotes a constant field. If it refers to a primitive type (they are boolean, byte, short, char, int, long, float, double in Java), it means the value of this field can not be changed. If it’s an Object, the value of this Object is still changeable, but you can not make this field refer to another thing. In short, the assignment statement is prohibited for a final field.
  • A final class can not be inherited, and a final method can not be overridden.
  • You can use the combo of them static final as a prefix.

[5] Constructor

  • Unlike a method, a constructor doesn’t have a return type.
  • The public Stack() is called default constructor, even if you leave it blank, JVM still assign all uninitialized fields with their default values, 0 for primitive numbers and char, false for boolean, null for Objects. The public Stack(Item item) is a parameterized constructor.
  • You can use any other constructors, such as this() or super(), inside a constructor. First of all, make sure such a constructor exists. Also, put any super() constructors before the this() ones.
  • Initialization Clause: You can put common initialization code inside a brace {}, between class fields and constructor. The following code initializes the demoList with an initialization block of an anonymous inner class. The common way is to do it with List.of(elements...).

[6] Getter and Setter

Usually, we set all class fields as private, to avoid accidentally change and malicious access from other classes. Sometimes, it’s necessary to change or get a field from outside, so Java designs setter and getter. The size() function in this class is a getter. Sometimes we name them with the field name, e.g. setN() and getN().

[7] Equality Test

  • In total, there are 2 kinds of equality test, identity-oriented (with calling equals() method) and value-oriented (with equal sign ==). The former one tests whether 2 variables occupy the same memory address, while the later one focuses on comparing their value.
  • Primitive types: all primitive types are value-oriented compared. Be careful about equality between double types, sometimes the precision difference caused by hardware limitation results in an invalid program.
  • String: all String are immutable Objects in Java, it means we can not change the content of a String once it’s created. Treat each string as an object, use == to test the memory address equality, and use equals() for content equality test.

[8] Documentation

Documentation is extremely crucial to code management, inheritance and update. You can use an inline documentation (// follows documentation content) to explain simple points, and use a block documentation (as showed in the above code snippets) to illustrate a class, a method, or any other obscure points.
Roughly speaking, there are 3 kinds of documentation,

  1. Class Documentation: usually comes as a block documentation, right above the class to be explained. State the authorities, objectives and functionality of this class briefly.

  2. Method Documentation: block documentation as well. Below is the structure of a method documentation.

  3. Class Field Documentation or Code Block Documentation: explain the function of a class field, or the logic of a code block with inline documentation, in a pithy style.

Java documentation supports a variant of tags and html syntax. The code snippets in Method Documentation has displayed some of them, more example are as followed,

  • @author [author name]
  • {@code [code snippets]}
  • <i>italic</i>, <em>emphasis</em>, <a href="hyperlink">, etc.

[9] toString() Method

A class implement the toString() method is printable. In the code System.out.print(objectName), the JVM will automatically call the objectName.toString() method of this object, and replace the objectName with the return value of objectName.toString(). So this method should return a String contains essential values in this Object. Actually, all class should implement this method for convenient debug.

[10] Static Inner Class

Sometimes we need to hide one class inside another to either avoid name collision or benefit package management. Static inner class, or nested class comes into being under such requirement.

  • Only inner class can be declared as static.
  • Declare a static inner class as public, private, protected, or default, then you can call it through OuterClassName.InnerClassName under the access rule defined by the keyword.
  • Static inner class act exactly like a top-level class, so it’s not a member of the outer class. For example, static inner class can have static features. Also, it has no privilege about accessing to outer class’s feature.
  • An interface’s inner class are automatically static and public.

[11] Exceptions

This is the hierarchy of Java Exception,

  • All Java Exceptions implement the Throwable interface. Error usually arise from internal errors or resource exhaustion, for which a programmer could do a little. In most cases, we don’t need to handle an Error in our program.

Throw

  • For a potential exception, we can either throw it or catch it. The above demo code shows how to throw an exception. First, throws it explicitly in the method header, then throw it in the method body. In method header, use comma to separate multiple exceptions.
  • If you override a method of a superclass, then the exceptions that subclass declares cannot be more general than those of the superclass. In other words, you may throw same exceptions, more specific exceptions, or throw no exception.
  • When creating an instance of exception, the String Stack underflow inside constructor NoSuchElementException() is the note to be displayed when this exception is triggered. A blank String means to use the default exception description.

Catch

  • The above code snippets show the traditional way to catch and handle an Exception.
  • If an exception occurs in the try clause, the JVM will skip the remaining codes, go through the corresponding catch clause and finally clause. If no such a catch clause, it throws the exception and terminates the program. If no exception occurs in the try clause, the JVM will skip all catch clauses, go for the finally clause to clean up resource.
  • You can catch and handle each Exception in one clause, or catch many of them (Use | to separate Exceptions inside the catch parentheses) and handle them together in one clause.
  • You can use the try(ResourceClassName resource1 = new ...; ResourceClass resource2 = new ...) syntax to initiate all necessary resources in the try clause, then all these resources is going to be closed automatically by JVM in the end, and you don’t have to write the finally clause explicitly.
  • So when to catch, and when to throw? Generally speaking, catch those you know how to handle, and throw away that you do not know how to handle.

[12] Inner Class

Previously we talked about static inner class, which aims at class hidden and package management. Now let’s walk through some details about the regular inner class. In the following paragraphs, I leave out the adjective “Regular”.

  • Inner class can access everything inside the instance that creates it, including all private features.
  • Inside the inner class, you can directly call the features of the outer instance. You can also use the syntax OuterClass.this.featureName to call them explicitly.
  • Inside the outer class, you can access all features of the inner class with their name directly, e.g. innerInstance.featueName.
  • If the inner class is accessible from the outside of the outer class (for example, when it’s public or default), use outerInstance.new InnerClass() to create an inner class instance, OuterClass.InnerClass to refer to the inner class, and outerInstance.innerInstance.featureName to call its features.
  • Inner class can have static fields, but they have to be final as well. However, static method for inner class is prohibited, as the complexity overweights the benefit.

[13] Iterator

  • A Class implements the Iterable interface if it has a public method iterator() that returns a class that implements the Iterator interface.
  • The Iterator interface contains 3 methods to be implemented,
  1. boolean hasNext(): whether there exist another element
  2. Class next(): return the next element
  3. void remove(): remove the last element returned by the call next()
  • In summary, if you want to develop an iterable class, first implement the Iterable interface on it, that is to add a public Iterator iterator() method. Next, design an inner class which implements the Iterator interface.
  • An iterable class can use the abbreviate for loop: for (IterableClass element: iterableCollection) to go through all elements inside iterableCollection, either ordered or unordered.

[14] Command Line Arguments

  • The command java ClassName will call the main method inside this class. Notice that all main method have the same arguments String[] args. This is the String list of command line arguments. More specifically, if you run java ClassName arg0 arg1 arg2, then inside the main method body, args[0] is the String format of arg0, args[1] is the String format of arg1, etc..
  • Due to this syntax, all command line arguments following the ClassName are considered as main method arguments. Consequently, all other functional command line arguments, such as --classpath, have to be placed before the ClassName.
