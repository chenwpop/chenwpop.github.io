Maven Tutorial

Author: Chen Wahng Date: Jul 05, 2019  19:53:15 Category: Java

Introduction

Maven, a Yaddish word meaning accumulator of knowledge. It’s a Java project management and build tool, which defines a clear statement of what a project consists of, provides an easy way to publish project information and share packages across multiple projects.

Objectives

providing a uniform build system

Maven allows a project to build using its project object model (POM) and a set of plugins that are shared by all projects using Maven, provide a uniform build system. Once you know how to build one Maven project, you automatically know how all other Maven projects build, consequently, saving you an immense amount of time when trying to navigate other projects.

providing quality project information

Maven provides plenty of project information, that is in part taken from your POM and in part generated from your project’s source files. For example, maven can provide,

  • Changelog document created directly from source control
  • Cross-referenced sources
  • Dependency list
  • Unit test reports

providing guideline for best practices development

Maven aims to guide a project in the direction for best practices development, as well as to assist in project workflows such as release and issue management.

Learn Maven in 5 Minutes

Structure of a Maven project

Here are some essential fields of a Maven project,

  • groupID: A unique identifier of the group or organization that develops this project, which is essentially the package name. A standard Java package name is the reverse order of a public domain, such as com.oracle, org.apacha.maven, in this way a package name can never collide with others. The package name also implies the structure of your package folder, for example, naming your package as com.oracle.app means your source files shall be placed in the path ./com/oracle/app/*.java. The Java compiler and Java Virtual Machine (JVM) will navigate your package and search for files according to this path, and Maven will automatically structure your package in this way. If someday, you decide to build a package from the ground, remember to put your files exactly in such a path.
  • artifactID: A unique name of the primary artifact being generated from your project. Consists of several words split by -, for example, wiki-app, app-parent, app-common. The primary artifact is usually a JAR file, named as [artifactID]-[version].[extension], e.g. wiki-app-1.0.jar.
  • pox.xml: Project Object Model. The core of project configuration in Maven.
  • src/main/java: Directory contains the project source code.
  • src/test/java: Directory contains the test code. For example, if you name your package with com.oracle.app, your source files and test files will be placed in src/main/java/com/oracle/app/*.java and src/test/java/com/oracle/app/*.java respectively.

Let’s walk through some essential tags,

  • project: The object of this project object model.
  • modelVersion: The version of this project object model.
  • packaging: Indicates the package type to be generated as the primary artifact (e.g. jar, war, ear, etc.).
  • version: The version of the artifact generated from this project. A suffix SNAPSHOT indicates current artifact is under development. Version without this suffix is considered as released, so the code is unmodifiable.

These tags are commonly used in Maven’s generated documentation,

  • name: The display name of this project.
  • url: The project’s site.
  • description: Provides basic description of your project.

Build the package

A simple command mvn package.

Notice that the package here is not a goal but a phase of Maven lifecycle, which is an ordered sequence of phases. When you run a command mvn [phase], Maven will execute every phase in the sequence up to and including the one specified.

Some important Maven lifecycles,

  • validate: validate the project is correct and all necessary information is available.
  • compile: compile the source files of the project.
  • test: compile the test files of the project to test the compiled sources (without packaging or deploying the code.)
  • package: package code in a distributable format, such as JAR.
  • verify: run any checks on results of integration tests to ensure quality criteria are met.
  • install: install the package into the local repository, for use as a dependency in other projects locally.
  • deploy: copy the final package to the remote Maven repository to share with other developers and projects.
  • clean: clean up artifacts created by previous builds.
  • site: generate site documentation for this project.

Add properties

Properties serve as the global variables in a pom, you can define any variable under the properties tag, usually, the property name is some keywords joint with dots. Additionally, The pom defines some properties naturally using the names of the XML elements that define the value, for example, ${project.name}$ refers to the display name of your project.
